# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A pipeline that generates LimaCharlie D&R rules from MITRE ATT&CK + Atomic Red Team data, deploys them, and reports coverage gaps. It also includes an AI correlation layer (Claude or Gemini) for multi-tactic attack detection.

## Pipeline Commands

Scripts run sequentially as numbered phases. Each reads the previous phase's output:

```bash
pip install -r requirements.txt

python 01_fetch_attack_data.py        # Downloads ATT&CK STIX + Atomic Red Team → data/
python 02_map_telemetry.py            # Maps data sources to LC events → mappings/technique_rules.yaml
python 03_generate_rules.py           # Generates D&R rules → output/rules/, output/all_rules.yaml
python 04_deploy_rules.py --dry-run   # Validates rules without deploying
python 04_deploy_rules.py             # Deploys to LC org via Hive API
python 05_coverage_report.py          # Gap analysis → output/coverage_report.md, output/navigator_layer.json
```

Deployment filters: `--filter-tactic execution`, `--filter-technique T1059`, `--clean` (delete existing rules first).

Credentials: `LC_OID` and `LC_API_KEY` env vars, or `limacharlie login` stored creds.

## Architecture

The pipeline is a linear 5-phase process. Each numbered script is standalone (no shared state beyond files on disk).

**lib/** contains the core logic, separated by concern:
- `attack_parser.py` — Parses ATT&CK STIX bundles into `Technique` and `DataComponent` dataclasses. Handles dual-bundle loading (latest STIX + older v15.1 for data component relationships).
- `atomic_parser.py` — Parses Atomic Red Team YAML tests, extracts indicators (paths, registry keys, IPs, domains, known tools, command patterns) via regex. `AtomicIndicator` and `AtomicTest` are the key dataclasses.
- `rule_generator.py` — Core rule generation. `INDICATOR_STRATEGY` dict maps indicator types to LC event types and detection approaches. `generate_rules_for_technique()` is the main entry point. Placeholder rule generation is disabled (`if False`).
- `lc_client.py` — LimaCharlie SDK wrapper. Deploys rules via `Hive("dr-general")`. Supports deploy, list, and delete-by-tag operations.

**Key mapping file:** `mappings/lc_event_to_datasource.yaml` is hand-authored and maps LC event types to ATT&CK data components. This is the bridge between the two systems and the file most likely to need manual updates.

**AI correlation:** `ai/correlation_prompt.md` is the prompt used by LC AI Sessions to correlate detections across a host when 2+ ATT&CK tactics appear within 24h. Two deployment options exist in `ai/` (AI Sessions via Claude, AI Agent Engine via Gemini).

## Rule Conventions

- Rule names: `attack-{TECHNIQUE_ID}-{short-slug}` (e.g., `attack-T1059.001-powershell-execution`)
- Every rule has a `report` action with structured ATT&CK metadata and a per-sensor suppression block
- Priority is calculated from a base (default 3) + threat group usage + privilege requirements, capped at 10
- Confidence levels: low (event-type-only), medium (indicator-based, default), high (tuned)
- Tags: `attack-coverage`, tactic name, `auto-generated`

## Gitignored Outputs

`data/` (ATT&CK/Atomic downloads), `output/` (generated rules/reports), and `mappings/technique_rules.yaml` are all regenerated by the pipeline. Only `mappings/lc_event_to_datasource.yaml` is checked in.
